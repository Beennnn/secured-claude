# ADR-0016: Supply-chain provenance — cosign keyless OIDC + Syft SBOM + pinned digests

- **Status**: Accepted
- **Date**: 2026-04-29
- **Deciders**: Benoit Besson

## Context

[OWASP A08:2021 Software & Data Integrity Failures](https://owasp.org/Top10/A08_2021-Software_and_Data_Integrity_Failures/) is the category covering compromised artifacts, malicious updates, and lack of integrity verification. For an enterprise pitch, the security review will explicitly ask :

- "How can a downstream consumer verify the image they pulled is the one your CI built ?"
- "What's in the image — is there a SBOM ?"
- "What if your registry is compromised ?"
- "What if a maintainer is compromised ?"

The answer cannot be "trust us." It must be cryptographic.

## Decision

We adopt a **3-pillar supply-chain provenance** :

### 1. Cryptographic signing — cosign keyless OIDC

Every container image released is signed with [cosign](https://github.com/sigstore/cosign) in **keyless mode** using GitLab CI's OIDC ID tokens as the signing identity. **No long-lived signing keys to manage** — the signing identity IS the GitLab CI job at a specific commit, recorded permanently in the [Sigstore Rekor transparency log](https://docs.sigstore.dev/logging/overview/).

CI signing snippet (in `.gitlab-ci.yml`, publish stage) :

```yaml
publish:cosign-sign:
  image: gcr.io/projectsigstore/cosign:v2.4.1@sha256:...
  variables:
    COSIGN_EXPERIMENTAL: "1"
  id_tokens:
    SIGSTORE_ID_TOKEN:
      aud: sigstore
  script:
    - cosign sign --yes ${CI_REGISTRY_IMAGE}/claude-code:${CI_COMMIT_TAG}
  rules:
    - if: $CI_COMMIT_TAG
```

User-side verification :

```bash
cosign verify \
  registry.gitlab.com/benoit.besson/secured-claude/claude-code:v0.1.0 \
  --certificate-identity-regexp '^https://gitlab.com/benoit.besson/secured-claude' \
  --certificate-oidc-issuer https://gitlab.com
```

This fails if : (a) the image was not signed by our CI, (b) the signature is invalid, (c) the signing certificate has been revoked, (d) the Rekor entry doesn't match.

### 2. Software Bill of Materials — Syft SPDX

Every release ships an SBOM in [SPDX 2.3](https://spdx.dev/use/specifications/) format, generated by [Syft](https://github.com/anchore/syft). Three SBOMs per release :

- `secured_claude-X.Y.Z-py3-none-any.whl.spdx.json` — Python wheel deps
- `claude-code-vX.Y.Z.spdx.json` — Container image (full OS-level + npm deps)
- `secured-claude-vX.Y.Z-aggregate.spdx.json` — Combined view

Generation in CI publish stage :

```yaml
publish:sbom:
  image: anchore/syft:v1.18.1@sha256:...
  script:
    - syft packages registry:${CI_REGISTRY_IMAGE}/claude-code:${CI_COMMIT_TAG} -o spdx-json=claude-code.spdx.json
    - syft packages dist/secured_claude-*.whl -o spdx-json=wheel.spdx.json
    - syft packages dir:. -o spdx-json=aggregate.spdx.json
  artifacts:
    paths: ['*.spdx.json']
```

SBOMs are attached to the GitLab Release as assets and can be cross-referenced with [grype](https://github.com/anchore/grype) for vulnerability scanning :

```bash
grype sbom:claude-code-v0.1.0.spdx.json
```

### 3. Pinning by digest, end-to-end

Per [ADR-0008](0008-pin-upstream-images-and-deps.md), every upstream image and dep is pinned by digest, not just tag. This means the supply chain attack surface is :

- Every dep version has been at some point reviewed and merged via Renovate PR
- A compromised upstream cannot silently propagate

## Consequences

### Positive

- **No long-lived signing keys to manage** — keyless OIDC means no `cosign.key` secret to leak ; the signing identity is verified against Sigstore's transparency log.
- **Forensically verifiable** — anyone can inspect the Rekor entry for our signing event :
  ```bash
  cosign tree registry.gitlab.com/benoit.besson/secured-claude/claude-code:v0.1.0
  ```
- **SBOM enables consumer-side scanning** — a consumer can run `grype sbom:claude-code.spdx.json` against the latest CVE feed without re-pulling the image, useful for air-gapped review.
- **Aligns with [SLSA v1.0 Build Level 2+ requirements](https://slsa.dev/spec/v1.0/levels)** — provenance records who built and what was built.
- **OWASP A08, SLSA, Sigstore** — three frameworks that any security review will check for.
- **Renovate auto-bumps with awareness** — any digest change requires a PR ; reviewers see exactly what shifted.

### Negative

- **Sigstore dependency** — if Sigstore (Rekor / Fulcio) goes down, signing fails. Mitigated by : (a) Sigstore is operated by the [Linux Foundation](https://www.sigstore.dev/), (b) verification can be done offline against a cached Rekor entry, (c) in extremis, fall back to detached signatures with our own key.
- **Verification is opt-in** — users who don't run `cosign verify` won't notice tampering. Mitigated by : (a) `secured-claude doctor` will verify in v0.2, (b) install scripts in v0.2 will refuse to proceed if verification fails.
- **SBOM size** — typical full-OS SBOM is 1-3 MB. Acceptable as release asset.
- **CI complexity** — adds three jobs to the publish stage (cosign sign, syft SBOM, attach to release). Mitigated by tested examples in the GitLab + Sigstore docs.

### Neutral

- We accept that v0.1 doesn't sign Python wheels (Sigstore for PyPI is still maturing). Tracked v0.2 once `pip install --require-hashes --verify-sigstore` lands officially.

## Alternatives considered

- **No signing, only digest pinning** — pinning protects against rebuilds but not against a registry MITM or compromised registry. Cosign closes that gap. Rejected as insufficient.
- **GPG-signed images** — older approach (Notary v1 / Docker Content Trust). Less ecosystem support than Sigstore today, requires private-key management. Rejected.
- **Notary v2** — successor to Notary, but adoption is lower than Sigstore. Tracked as fallback option.
- **Self-hosted Rekor** — overkill for v0.1 ; Sigstore public is fine.
- **No SBOM** — fails OWASP A06 / NIST SSDF PS.2. Rejected.
- **CycloneDX format SBOM instead of SPDX** — both are widely supported. SPDX picked for ISO 5962 standardization. CycloneDX exported alongside in v0.2 if requested.
