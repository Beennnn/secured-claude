# 18. Package version derived from git tag (hatch-vcs)

Date: 2026-04-29
Status: Accepted

## Context

In v0.1.x the wheel version was hardcoded in `pyproject.toml` (`version = "0.1.0"`). Every `vX.Y.Z` git tag produced a wheel with the same filename — `secured_claude-0.1.0-py3-none-any.whl` — which the GitLab Package Registry rejected with HTTP 400 Bad Request on the second tag attempt:

```
HTTPError: 400 Bad Request from https://gitlab.com/.../packages/pypi
A file with that name already exists in the package registry.
```

GitLab's twine adapter does not implement `--skip-existing` (returns `UnsupportedConfiguration`), so the v0.1.x publish:pypi job carried a **shell-wrap shim** that caught "400 Bad Request" / "already exists" and treated it as success for idempotent re-tags. This was tagged as a v0.2 exit ticket in [ADR-0014](0014-gitlab-ci-pipeline-6-stages.md) and the publish.yml comment.

The shim worked but had three downsides :

1. **Hides real failures** — any twine error that mentions "400" (auth issues, malformed metadata) would be silently treated as success.
2. **Same-version artefacts diverge silently** — if the wheel content drifted between v0.1.7 and v0.1.8 (it did not, but could), the registry kept the v0.1.7 wheel and refused the v0.1.8 one. Verifiers downloading "v0.1.8" got bytes from v0.1.7.
3. **No audit trail per tag** — `pip install secured-claude==0.1.7` and `pip install secured-claude==0.1.8` both resolve to the same physical wheel.

## Decision

Adopt **hatch-vcs** to derive the package version from the latest reachable git tag at build time. `pyproject.toml` declares `dynamic = ["version"]` and the `[tool.hatch.version]` source is `vcs`, so :

- `git tag -a v0.2.0` then `uv build` → `secured_claude-0.2.0-py3-none-any.whl`
- `git tag -a v0.2.1` then `uv build` → `secured_claude-0.2.1-py3-none-any.whl`
- Untagged dev commit at HEAD → `secured_claude-0.2.1.dev3-py3-none-any.whl` (3 commits past v0.2.0 toward the next 0.2.1)

Each tag produces a **unique wheel filename**. The publish:pypi shell-wrap shim is removed ; plain `twine upload dist/*` is now intrinsically idempotent across re-tags (a re-tag of the same SHA produces the same filename, which the registry refuses with a 400 — but a NEW tag always uploads cleanly).

`__version__` in `secured_claude/__init__.py` reads from the auto-generated `_version.py` (gitignored, written by hatch-vcs at build time), with a fallback to `0.0.0+unknown` for development checkouts that haven't been built.

`raw-options` configure setuptools_scm-compatible behaviour :
- `version_scheme = "guess-next-dev"` — untagged commits get a `.devN` suffix, never collide with the previous tag's version.
- `local_scheme = "no-local-version"` — strip the `+gabc1234` local version that PyPI / GitLab Package Registry reject.

The CI build:python job now runs `apt-get install -y git` in `before_script` (the `uv:python3.13-bookworm-slim` base image lacks `git`, which hatch-vcs requires), and `git fetch --tags --depth=100` to ensure the tag is reachable in the shallow GitLab CI clone.

## Consequences

**Positive** :
- Each release ships a uniquely-named wheel ; verifiers checking `secured_claude-0.2.0-py3-none-any.whl` get the v0.2.0 bytes deterministically.
- The publish:pypi shim is gone — real twine errors propagate with their actual message instead of being masked.
- Local dev builds (`uv build` without a tag) get a `.devN` version that pip resolves correctly without colliding with releases.
- Standard pattern across the Python ecosystem — every audit / RSSI / dependency reviewer recognises hatch-vcs.

**Trade-offs** :
- build:python job adds ~3 s for `apt-get install git` (acceptable ; the network/cache cost outside this job is in the multi-second range anyway).
- The `_version.py` file is auto-generated and must be in `.gitignore` — a developer who manually creates it would shadow the build hook. Mitigated by the comment in the file itself : `# file generated by vcs-versioning / don't change, don't track in version control`.

**Risks** :
- A CI job that runs `uv build` without `git` available (or in a shallow clone with no tags) silently produces a `0.0.0+unknown` wheel. Mitigated by the `git fetch --tags` step in build:python and the import fallback in `__init__.py`.

## Verification

- `uv build` at HEAD with no tag matching → produces `secured_claude-0.X.Y.devN-py3-none-any.whl`
- `git tag v0.2.0-rc1 && uv build` → produces `secured_claude-0.2.0rc1-py3-none-any.whl`
- `git tag v0.2.0 && uv build` → produces `secured_claude-0.2.0-py3-none-any.whl`
- `python -c "import secured_claude; print(secured_claude.__version__)"` matches the wheel filename version

## Alternatives rejected

- **Keep hardcoded version + manual bump per tag** — every release requires bumping `pyproject.toml` AND tagging, doubles the change surface, easy to forget. Was the v0.1 default ; explicitly out for v0.2.
- **setuptools-scm directly** — works but our build backend is hatchling, and hatch-vcs is the hatchling-native bridge (it wraps setuptools-scm internally). Same reliability, less plumbing.
- **CI-injected version via `SETUPTOOLS_SCM_PRETEND_VERSION`** — works for tag builds but breaks local `uv build` and PR pipelines. The git-based scheme covers both with no env-var dance.
