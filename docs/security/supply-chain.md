# Supply chain security — secured-claude v0.1

> **Mission** — every artifact a user installs (Python wheel, Docker image, install script) must be **traceable to a verified source**, **integrity-checkable**, and **reproducible**. We follow [SLSA v1.0 framework](https://slsa.dev/) goals and align with [OWASP A08:2021 Software & Data Integrity Failures](https://owasp.org/Top10/A08_2021-Software_and_Data_Integrity_Failures/).

---

## 1. Pinning strategy

| Artifact type | Pinning level | Why |
|---|---|---|
| **Docker images** (Cerbos, Node base, runners) | Image digest `@sha256:...`, NOT just tag | Tags are mutable ; digests are content-addressed |
| **Python deps** (runtime + dev) | Exact version in `pyproject.toml` + locked digest in `uv.lock` | Reproducible installs, no silent transitive bumps |
| **npm deps in agent container** | Exact version in `package.json` + integrity hash in `package-lock.json` | Same |
| **OS packages in Dockerfile.claude-code** | `apt-get install <pkg>=<version>` (Debian-pinned where feasible) | Avoid silent OS-package upgrades on rebuild |
| **CI runner images** | Pinned tag + image digest | Avoid silent runner upgrades breaking pipelines |
| **GitLab CI image references** | `image: foo:1.2.3@sha256:abc...` | Same |
| **Install scripts** (`install.sh`, `install.ps1`) | Sourced from canonical URL with HTTPS+SHA verification | Reproducible installs |

**No `:latest` tags anywhere.** This is enforced by `hadolint` rule DL3007 in CI lint stage.

**No floating Python dep versions** (e.g. `requests>=2.32`) without `uv.lock` to pin the actual version. CI verifies `uv.lock` is in sync with `pyproject.toml` via `uv lock --check`.

## 2. Software Bill of Materials (SBOM)

Every release ships an **SBOM in SPDX 2.3 format**, generated by [Syft](https://github.com/anchore/syft).

### 2.1 What we publish

For each `vX.Y.Z` release, the GitLab Release page contains :

- `secured_claude-X.Y.Z-py3-none-any.whl.spdx.json` — Python wheel SBOM
- `claude-code-vX.Y.Z.spdx.json` — Container image SBOM (full OS-level deps)
- `secured-claude-vX.Y.Z-sbom-aggregate.spdx.json` — Combined view

### 2.2 Generation in CI

```yaml
# .gitlab-ci.yml fragment (publish stage)
publish:sbom:
  image: anchore/syft:v1.18.1@sha256:...
  script:
    - syft packages dir:. -o spdx-json=secured-claude-aggregate.spdx.json
    - syft packages registry:$CI_REGISTRY_IMAGE/claude-code:$CI_COMMIT_TAG -o spdx-json=claude-code.spdx.json
    - syft packages dist/secured_claude-${CI_COMMIT_TAG#v}-py3-none-any.whl -o spdx-json=wheel.spdx.json
  artifacts:
    paths: ['*.spdx.json']
```

### 2.3 How users verify

```bash
# Download SBOM from GitLab Release
curl -O https://gitlab.com/benoit.besson/secured-claude/-/releases/v0.1.0/downloads/claude-code.spdx.json

# Inspect with grype (cross-references with vulnerability DB)
grype sbom:claude-code.spdx.json

# Or with syft directly
syft sbom:claude-code.spdx.json
```

## 3. Cryptographic signing

### 3.1 Container images — cosign keyless OIDC (Sigstore)

Every container image is signed via [Sigstore cosign](https://github.com/sigstore/cosign) using the **keyless** mode tied to GitLab CI's OIDC ID token. **No long-lived signing keys to manage** — the signing identity is the GitLab CI job itself, verifiable by anyone via the Rekor transparency log.

#### CI signing snippet

```yaml
# .gitlab-ci.yml fragment (publish stage)
publish:cosign-sign:
  image: gcr.io/projectsigstore/cosign:v2.4.1@sha256:...
  variables:
    COSIGN_EXPERIMENTAL: "1"
  id_tokens:
    SIGSTORE_ID_TOKEN:
      aud: sigstore
  script:
    - cosign sign --yes $CI_REGISTRY_IMAGE/claude-code:$CI_COMMIT_TAG
  rules:
    - if: $CI_COMMIT_TAG
```

#### User-side verification

```bash
cosign verify \
  registry.gitlab.com/benoit.besson/secured-claude/claude-code:v0.1.0 \
  --certificate-identity-regexp '^https://gitlab.com/benoit.besson/secured-claude//.gitlab-ci.yml@.*' \
  --certificate-oidc-issuer https://gitlab.com
```

This fails (non-zero exit) if :

- The image was not signed by our CI
- The signature is invalid
- The signing certificate has been revoked

### 3.2 Git tags — SSH-signed by maintainer

```bash
git config user.signingkey ~/.ssh/id_ed25519.pub
git config gpg.format ssh
git tag -s vX.Y.Z -m "..."
git verify-tag vX.Y.Z
```

The maintainer's public SSH key is published in [`docs/security/maintainer-keys.md`](maintainer-keys.md) (created when first key is registered). CI on `main` will reject any unsigned tag (rule TBD in v0.1, mandatory v0.2).

### 3.3 Python wheels — GitLab Package Registry + SHA256

Wheels uploaded to GitLab Package Registry get an automatic SHA256 sum visible at the registry page. v0.2 will add Sigstore signing for Python (`pip install secured-claude --require-hashes`).

## 4. Reproducibility

We aim for **bit-for-bit reproducible builds** where the toolchain allows :

- **Python wheels** built with [Hatchling](https://hatch.pypa.io/latest/) — Hatchling produces deterministic wheels given a fixed source tree.
- **Container images** : `Dockerfile.claude-code` uses pinned base image digest + apt-get pinned versions. The build CAN diverge if upstream moves (mitigated by digest pin).
- **`uv.lock`** committed to repo — `uv sync` produces identical environments.

### 4.1 Verification by reproducing

```bash
git clone https://gitlab.com/benoit.besson/secured-claude.git
cd secured-claude
git checkout v0.1.0
uv sync
uv build
sha256sum dist/secured_claude-0.1.0-py3-none-any.whl
# Compare to the SHA256 published in the GitLab Release notes
```

For the container image :

```bash
docker buildx build --platform linux/amd64 -t secured-claude/claude-code:repro -f Dockerfile.claude-code .
docker image inspect secured-claude/claude-code:repro | jq '.[0].Id'
# Compare to the digest published in the Release notes
```

(In practice, container reproducibility across machines is hard due to base-image refresh ; we publish the digest at *build time*, not as a reproducibility guarantee.)

## 5. Dependency update policy (Renovate)

[`renovate.json`](../../renovate.json) configuration :

- **Auto-merge** : patch + minor bumps for trusted ecosystems (PyPI, npm, Docker Official Images) IF CI is green
- **Manual review** : major bumps (semver), new transitive deps, base image refreshes
- **Group rules** :
  - All `cerbos/cerbos:*` digests grouped (single PR)
  - All Python deps in one PR per week (Tuesday)
  - All Docker base image digests in one PR (digest-only updates)
- **Schedule** : business hours UTC (avoids 3am wake-ups)
- **Auto-close** : security-only updates auto-merged ASAP regardless of schedule

Sample rules (full file in repo) :

```json
{
  "extends": ["config:recommended", "group:dockerDigests"],
  "packageRules": [
    {
      "matchDepTypes": ["devDependencies", "dev"],
      "automerge": true
    },
    {
      "matchPackageNames": ["cerbos/cerbos"],
      "groupName": "cerbos pdp"
    },
    {
      "matchUpdateTypes": ["major"],
      "automerge": false,
      "addLabels": ["needs-review"]
    }
  ],
  "vulnerabilityAlerts": {
    "labels": ["security"],
    "automerge": true
  }
}
```

## 6. License compliance

We track all dependency licenses to avoid accidentally pulling in restrictive ones. CI runs [pip-licenses](https://github.com/raimon49/pip-licenses) (or syft license summary) and FAILS if a non-allowlisted license appears.

**Allowlisted licenses** :

- MIT, BSD (2-clause, 3-clause), Apache 2.0, ISC, MPL 2.0, Python Software Foundation, Public Domain (Unlicense / CC0)

**Reviewed manually** (require ADR if introduced) :

- LGPL 2.1+, EPL 2.0, GPL with classpath exception

**Refused** :

- GPL 2.0/3.0 (incompatible with our MIT permissive license unless dynamic linking)
- AGPL 3.0 (incompatible)
- Commercial / non-free

The license report is published as a release asset (`secured-claude-vX.Y.Z-licenses.json`).

## 7. CI security on the supply chain itself

The CI pipeline IS part of the supply chain. We protect it by :

- **GitLab CI/CD variables** : group-level for credentials, marked `protected` (only on protected refs `main`, tags) and `masked` (redacted in logs)
- **`CI_JOB_TOKEN`** scoped to current job — cannot be exfiltrated to read other projects
- **OIDC ID tokens** for cosign signing — short-lived, audience-bound
- **Renovate token** read-only on the project — cannot create unrelated PRs
- **No GitHub Actions** for the canonical pipeline — only a passive miror push (CLAUDE.md global rule)
- **Branch protection on `main`** : direct push denied, merge requires green pipeline + maintainer approval

## 8. Verification commands cheatsheet

For users who want to verify before installing :

```bash
# 1. Cosign signature on the Docker image
cosign verify \
  registry.gitlab.com/benoit.besson/secured-claude/claude-code:vX.Y.Z \
  --certificate-identity-regexp '^https://gitlab.com/benoit.besson/secured-claude' \
  --certificate-oidc-issuer https://gitlab.com

# 2. SBOM of the image (with grype, cross-checks against CVE DB)
syft packages registry:registry.gitlab.com/benoit.besson/secured-claude/claude-code:vX.Y.Z -o spdx-json | grype sbom:-

# 3. Python wheel SHA256
sha256sum ~/.local/share/secured-claude/cache/secured_claude-X.Y.Z-py3-none-any.whl
# Compare to the value at https://gitlab.com/benoit.besson/secured-claude/-/releases/vX.Y.Z

# 4. Git tag signature
git verify-tag vX.Y.Z

# 5. Source tree license summary
pip-licenses --from=mixed --format=json
```

## 9. Residual supply-chain risks (acknowledged)

| ID | Risk | v0.1 mitigation | v0.2+ plan |
|---|---|---|---|
| R-SC-1 | Compromised `cerbos/cerbos` upstream image | Digest pinning + Renovate rapid-bump on Cerbos security advisory | Run our own Cerbos build (less likely but possible) |
| R-SC-2 | Compromised `node:22-slim` base | Digest pinning, hadolint scan, trivy scan in CI | Switch to a minimal-distroless base for production |
| R-SC-3 | Compromised PyPI dep (typosquat, account hijack) | uv.lock + pip-audit + Renovate vuln alerts | Sigstore-signed wheels (when ecosystem matures) |
| R-SC-4 | GitLab Package Registry compromise | Cosign on images mitigates ; wheels signed in v0.2 | Out of our control beyond what GitLab provides |
| R-SC-5 | Renovate auto-merge merges a malicious patch | Auto-merge requires CI green = audit-demo --strict pass = wouldn't help if the dep itself isn't tested | Run `cargo-vet`-equivalent audit lookup for Python in v0.2 |
